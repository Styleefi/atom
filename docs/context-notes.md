# Context notes — decisions and reasoning

Append-only log so any later session (human or agent) can pick up without re-deriving decisions. Newest entries at the bottom.

## 2026-07-19 — foundation design (structured interview)

- **Goals were re-derived from real pains, not kept from the old README.** Six recurring pains were collected (corrections trapped in memory, setup repetition, unverified "done", lost session context, overengineering, config fragmentation across home/work machines). The old "Core Objectives" were rewritten into the 5 pain-driven goals now in README. Agent Pipelines survived because a real, positive experience existed (implement → arch-review → test-runner → doc-writer), not on aspiration.
- **Key insight driving goal 3:** pains 3/4/5 recurred *despite* rules already written in the personal global CLAUDE.md — written advice alone does not change behavior; deterministic enforcement (hooks, checkers, CI) is required. Hence the first artifact with teeth is the rules checker, and it runs in CI so the checker itself is not optional.
- **Rule system:** `meta/rules/` is the SSOT; a rule counts only when deployed to one of three vessels (claude-md / skill / hook), declared via frontmatter. Undeployed rules are worse than none (illusion of enforcement).
- **Propagation is pure git.** Canonical config lives inside the repo; children clone and pull from upstream. Symlinks and `~/.claude/` global placement were considered and rejected (per-machine setup cost, unwanted global side effects, name-conflict risk). The user identified project-scoped `.claude/` + clone as sufficient — design simplified accordingly.
- **Clone recipe, not fork/template:** GitHub cannot fork into the same account; "Use this template" severs history and kills `git pull upstream main`.
- **Personal `~/.claude/CLAUDE.md` is out of scope.** Atom is responsible only for its children. Do not touch the personal global file until legacy rules (§1–11) are migrated into `meta/rules/` (it is the only live copy until then); afterwards slim it to avoid double loading.
- **Role catalog and pipelines intentionally undecided.** The 8 home-grown roles are explicitly *not* presumed to be the answer; focused review tracked in Issues. `.claude/agents/`·`.claude/skills/` are not created until content is decided (no speculative structure).
- **Folder layout:** meta layer isolated in `meta/` so child projects' roots stay clean for their product layer; fixed-path files (root CLAUDE.md, `.claude/`, `.mcp.json`) stay put; one subpackage per harness with its own `tests/` (+`__init__.py` against pytest name collisions).
- **Stack:** Python >= 3.12 + pytest for the meta layer only (local 3.14.4 verified); child product stacks are declared per-project via the template's Toolchain section. Codified as the `python-stack` rule in PR 2.
- **Backlog lives in GitHub Issues** (not a committed BACKLOG.md) so atom's own to-dos do not pollute child projects; committed docs/ session artifacts are cleared by children at bootstrap (template instructions).
- **Documents in English; commits English (Conventional Commits); conversation and code comments Korean** (per personal global guidelines).

## 2026-07-19 — PR 2 (rules + checker) decisions

- **Checker strengthened (owner decision):** verification-less passing is forbidden. `enforce: hook`/`skill` rules are rejected until their deployment verification is implemented. For `claude-md`, the `@meta/rules/<file>` import line in the target *is* the deployment (imports auto-load every session), so the checker verifies real deployment, not just formatting.
- **`plan-deviation` rule born from a live mistake:** an unplanned `uv` install was attempted without presenting options; corrected and codified the same day. Deviations from an approved plan require situation + options + trade-offs + owner decision before execution.
- **Environment isolation (owner question exposed a real flaw):** the root `pyproject.toml` would have collided with a Python child project's product `pyproject.toml`, mixing meta and product deps in one env (pytest version conflicts). Fixed: `meta/` is a self-contained uv project (own `pyproject.toml`, committed `uv.lock`, own `.venv`); the repo root carries no Python project. venv (manual, no lockfile) and Docker (overkill) were compared and rejected; uv chosen (no sudo, lockfile reproducibility across home/work, CI parity via setup-uv).
- **Version floor policy:** floors at the current major (`pytest>=9.0`, `PyYAML>=6.0`), no caps — installs always resolve latest; the floor is a compatibility claim, not an install pin. Runtime messages (print/exceptions) in English; comments/docstrings Korean.
- **`answer-first` rule born from repeated live corrections:** answers were substituted by other artifacts — commits bundled into the same turn, and plan-approval dialogs shown instead of a direct reply. The rule generalizes to "never substitute the answer": prose answer first, execution (including plan approval) only after the owner responds. Vessel review for both behavioral rules (plan-deviation, answer-first): `skill` fails because the violator does not recognize the need for guidance at the violation moment; `hook` fails because the violations are intent-level, not machine-detectable (a question may legitimately require tool use to answer). Both are inherently always-relevant interaction protocol → `claude-md`, kept terse; the claude-md budget is guarded via Issue #3 and a possible future checker size limit.

## 2026-07-19 — PR 3 (issue workflow + first hook) decisions

- **Issue workflow split into two rules by enforcement nature** (owner pushed back on defaulting to claude-md for lack of checker support; verification implementation was pulled into scope instead of excusing the choice). Judgment-based conventions (find work via issue tracker, `Closes #n`, context in issue comments) → `issue-workflow` (claude-md). The one mechanically enforceable piece — search before create — → `issue-duplicate-guard` (hook), the first hook rule. `skill` rejected on pure fit: probabilistic loading fails exactly at the moment that matters.
- **Hook vessel convention fixed (resolves the open half of #4):** deployed-to is the settings JSON; it must reference `harness.<rule-id with - → _>` in a hook command, and that harness package must exist. Substring-level check is explicit v1.
- **Dual-platform requirement confirmed:** some children already run self-hosted GitLab CE (Docker); GitHub has no free self-host (GHES is paid), so GitLab stays. gh and glab are peers in rules and guard; remaining GitHub coupling (CI, README recipe) tracked in #10; glab adapter real-instance verification in #9 (no glab CLI on the dev machine; a fresh GitLab Docker was rejected as disproportionate to verifying a few CLI flags).
- **Guard design invariants:** (1) no false blocks — tokenize the whole command first (shlex, punctuation_chars) so quoted mentions can never sit in command position; (2) every failure path fails open (the hook runs on every Bash call and must never brick the tool) — including a `command -v uv` shell-level guard for machines without uv; (3) any wrong block is recoverable in one retry: the `ATOM_DUP_REVIEWED=1` override doubles as the false-positive escape hatch. Marker renamed from `ATOM_DUP_OK` (owner: unclear what is being OK'd) to state the actual claim — candidates were reviewed.
- **CRLF incident:** the entire working tree had been silently flipped to CRLF by a Windows-side tool (WSL), showing 20 phantom-modified files. Diagnosed with `git diff --ignore-cr-at-eol` (empty → content-identical), restored from index, and `.gitattributes` (`* text=auto`) added so it cannot recur.
